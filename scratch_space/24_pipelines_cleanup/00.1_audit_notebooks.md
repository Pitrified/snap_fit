# Audit - notebooks and scripts in scratch_space

Inventory taken 2026-07-21 to support [00_start.md](00_start.md).
Draft: numbers are measured, the judgements are a first pass.

## Method

Every `.ipynb` and `.py` under `scratch_space/` was measured mechanically
(code cells, non-comment code lines, inline `def`/`class` count, cells
importing `snap_fit`, longest cell, stored outputs) and each snap_fit import
block was executed to see whether it still resolves. Classification is a
reading pass on top of those numbers.

## Prior art

`scratch_space/20_piece_markers/13_pipelines_cleanup.md` already did an audit
of this exact kind: 27 notebooks + 1 script, classified PIPELINE / EXPLORATION
/ MIXED, with a proposed `pipelines/` tree and a migration mapping table
("Copy, clean up cell outputs, add markdown headers").

That inventory is still a useful reference and its capability grouping largely
holds. Its **plan** is superseded: it was a lift-and-shift of 13 existing
notebooks into `pipelines/`, which is exactly the "mesh of old things uplifted
randomly" this effort is trying not to produce. See D2 in
[00_start.md](00_start.md).

Deltas since that audit: `20_piece_markers/01_print_read_board.ipynb` and the
two `23_green_background/` scripts are new, and `16_support.py` was not
counted.

## Findings

- F1: most of these are development notebooks, and that is fine.
  They are code-heavy and drifting stale because that is what a prototype is
  for. Nothing here should be "fixed in place"; the audit is not a defect list.
  This is the reason the effort authors new pipelines instead of repairing old
  notebooks.
- F2: 37 inline `def`/`class` definitions live in notebook cells, concentrated
  in a few files (`weird_edge_shape_detection` 11, `naive_linear_solver/01` 5,
  `segment_id_model/01` 5, `sheet_manager/01` 4). Most are prototypes whose
  ideas already landed in `src/` (`SegmentId`, `SheetManager`, `PieceMatcher`).
  The residue that never landed is the interesting part, not the volume.
- F3: cell sizes are the clearest "this is not a pipeline" signal.
  A 442-line cell (`naive_linear_solver/01`) and a 216-line cell
  (`20_piece_markers/00_sample`) are programs, not loops. The target shape
  (flat loops over `src` calls) implies cells of roughly 5-20 lines.
- F4: `20_piece_markers/16_support.py` (268 LOC) is the clearest instance of
  the rule "useful code should not be in scratch_space". It re-derives the
  coordinate transforms between board / rectified / cropped spaces, which is
  library knowledge that `docs/guides/coordinate_spaces.md` documents but no
  module owns end to end. Promotion candidate.
- F5: the two `23_green_background/` scripts are already close to the target
  shape: thin, one job each, all logic in `src`, runnable, and validated on
  real captures. They are the best available template for what a pipeline
  should look like, and the strongest candidates to become the first two real
  pipelines.
- F6: import health is better than expected. 26 of 28 notebooks have snap_fit
  import blocks that still resolve. `aruco_setup/01_aruco_experiments` has no
  snap_fit imports at all (pure cv2). Importing is not running, so this is a
  floor, not a pass.
- F7: only one notebook uses an API that is now gone
  (`fastapi_scaffold/01_db_ingestion` reads `Sheet.threshold`, removed by the
  preprocess config refactor). Staleness is currently shallow, but nothing
  prevents it deepening, since no notebook is executed by anything.
- F8: no notebook stores cell outputs, so the repo is not carrying output
  bloat. A pre-commit hook already enforces this.
- F9: the same capability is covered several times over at different ages.
  Board generation exists in `aruco_setup/03`, `20_piece_markers/01`, and
  `23_green_background/generate_green_board.py`. Picking "the newest" per
  capability is a decision the audit can inform but not make automatically.

## Open points for the plan

- Which capabilities deserve a pipeline at all, versus staying documented-only.
- Whether `pipelines/` entries are notebooks, scripts, or both, given that some
  of these (generation, ingest) are already better as scripts.
- What "up to date" is enforced by, since nothing executes these today.
- What happens to `scratch_space/` afterwards: left as history, or pruned.
